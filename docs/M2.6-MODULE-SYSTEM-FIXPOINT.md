# M2.6 — `lib.nixosSystem` infinite-recursion in sui-eval

> Status: **OPEN**.  Captures the operator-blocking bug for sui as a
> drop-in `nixos-rebuild`/`darwin-rebuild` engine.  Surfaced by
> `sui-sweep --corpus rebuild` against real NixOS flakes; the M0
> shadow-testing surface (`sui-spec/specs/rebuild_probes.lisp`)
> records the probe that fails (`nixos-toplevel-outpath`,
> `nixos-dry-run-closure`, `nixos-closure-references`).

## TL;DR

`lib.evalModules` from nixpkgs (`lib/modules.nix`) works in sui-eval
for synthetic minimal modules.  `lib.nixosSystem`, which calls the
same `evalModules` with the full NixOS module set, hits infinite
recursion through `extraArgs ↔ matchedOptions` even when the
operator passes `modules = []`.

## Minimal repro

```sh
sui eval --no-vm --impure --raw '
let nixpkgs = builtins.getFlake "github:NixOS/nixpkgs/b77b3de";
in (nixpkgs.lib.nixosSystem { system = "x86_64-linux"; modules = []; })
   .config.system.name
'
```

Expected: a short string like `"nixos"`.
Today: `Eval(InfiniteRecursion(...))`.

## Force chain (16 frames)

Captured with the diagnostic improvement in commit-following-this-doc
(force-stack always populated, not gated on `SUI_TRACE_EVAL`):

```
→ let \n\n          # For definitions that have an associated opt    (lib/modules.nix)
→ mapAttrsRecursiveCond (v: !isOption v) (_: v: v.value) options     (lib/modules.nix)
→ mapAttrs (n: v: v.matchedOptions) resultsByName                    (lib/modules.nix)
→ mapAttrs (name: decls: # We're descending …)                       (lib/modules.nix)
→ zipAttrsWith (n: v: v) (map (module: …))                           (lib/modules.nix)
→ (reverseList (doCollect { }).modules)                              (lib/modules.nix)
→ length xs                                                          (lib/lists.nix)
→ filterModules modulesPath (collectStructuredModules unknownM…)     (lib/modules.nix)
→ keyFilter modules                                                  (lib/modules.nix)
→ concatMap ({ file, disabled }: map (moduleKey file) disabled)      (lib/modules.nix)
→ concatLists (catAttrs "disabled" modules)                          (lib/modules.nix)
→ (if module.disabledModules != [] then …)                           (lib/modules.nix)
→ checkModule (loadModule args parentFile "${parentKey}:anon-$")     (lib/modules.nix)
→ (loadModule args parentFile "${parentKey}:anon-${toString n}…")    (lib/modules.nix)
→ (applyModuleArgsIfFunction (toString m) (import …))                (lib/modules.nix)
→ <native-thunk>                                                     (<eval>)
```

The cycle bottoms out at a sui-internal `Native` thunk — i.e. a
`ThunkRepr::Native(f)` created by sui-eval (not parsed from Nix
source).  This thunk is the `extraArgs` value that
`applyModuleArgsIfFunction` is forcing to pass as the second argument
to each module function (`f args`).  The chain re-enters the top
because `extraArgs.<x>` depends on `config.<y>` which depends on the
module set's `options`, and the module set's options depend on
`extraArgs` having been resolved.

## Why minimal `evalModules` works but `nixosSystem` doesn't

The synthetic `lib.evalModules { modules = [...]; }` call we used as
a regression test passes a `modules` list whose attrs declare options
+ config DIRECTLY, with no `_module.args` dependency.  Sui-eval's
fix-point + blackhole detection handles that case correctly (see
`force_chain_captures_self_reference` and friends in `trace.rs`
tests).

`lib.nixosSystem` is structurally different: it bootstraps an
attrset of `extraArgs` (`{ inherit lib config options pkgs modulesPath …; }`)
where every field is a closure over the not-yet-computed module
result.  cppnix breaks this cycle by making each entry a SEPARATE
lazy thunk and forcing them only on first deep access; in particular,
`_module.args.pkgs` only forces when a module body actually selects
it.  Sui-eval today appears to eagerly force the full `extraArgs`
attrset somewhere in the call chain — probably in `args // extraArgs`
or in the `mapAttrs (name: _: addErrorContext context)` wrapping —
which collapses the fix-point.

## Suggested investigation paths

1. **Instrument `Native` thunk creation sites** — print a backtrace
   when sui-eval creates a `ThunkRepr::Native`.  The chain shows the
   cycle bottoms out at one of these; pinpoint *which* native-thunk
   construction is the bootstrap of `extraArgs`.

2. **Check `//` (attrset merge) operator** — `args // extraArgs` is
   the immediate predecessor in the chain.  If sui's `//` forces both
   operands' structure (key sets), it will demand-force a thunked
   `extraArgs` even when only `args`'s keys would suffice.  cppnix's
   `//` lazily merges; the result is an attrset whose KEY lookup
   defers forcing the source's individual values.

3. **Trace `_module.args` resolution** — that's the option whose
   value is the fix-point seed.  In nixpkgs `lib/modules.nix`,
   the `_module.args` declaration uses `lazyAttrsOf` with a custom
   merge function that's `lazy-aware`.  Sui's option-type registry
   (`sui-spec/src/module_system.rs::MergeStrategy`) lists `AttrsetMerge`
   for `attrs`, which deep-merges — is the LAZY-attrs variant
   correctly preserved?

4. **Compare against `builtins.sui.evalModules`** — sui's typed-Lisp
   bridge in `sui-eval/src/builtins/module_system_bridge.rs` already
   has an implementation of the algorithm.  If that bridge avoids the
   recursion (because it runs the fixed-point in Rust), there's an
   incremental-cutover path: detect `lib.evalModules` calls and
   redirect to the bridge.

## Second attempt — `SUI_BLACKHOLE_AS_NULL` bridge (env-gated, lands)

Tried: in the `ThunkRepr::Blackhole` arm of `force_inner`, return
`Value::Null` instead of `EvalError::InfiniteRecursion` when the
operator sets `SUI_BLACKHOLE_AS_NULL=1`.  Theory: cppnix exposes the
not-yet-complete value on re-entrance; we can't synthesize that
without a proper "partial promise" thunk variant, but a null
sentinel combined with `Select`'s NotAttrs-falls-to-default
behavior approximates it enough to let evaluation proceed.

Result: **the cycle breaks**.  Sui no longer raises
`InfiniteRecursion` on `lib.nixosSystem { modules = []; }`.  The
evaluation continues through `modules.nix`,
`system/boot/networkd.nix`, and bottoms out at:

```
TypeError("type error: cannot select from null
  while calling function defined in modules.nix
  while calling function defined in system/boot/networkd.nix
  …  while calling the 'concatLists' builtin
  while calling the 'concatMap<partial>' builtin
  while calling the 'elem<partial>' builtin
  while calling the 'genericClosure' builtin
  …")
```

This is downstream: some `nixos` module reads `x.y` where `x` is
the null sentinel (returned by the blackhole bridge), and the
access doesn't have `or default`.  cppnix never reaches this state
because cppnix's `x` would have been the not-yet-complete attrset,
which CAN be `.y`-selected (recurses lazily).

The bridge is shipped behind the env var (default OFF) so operators
can experiment.  Default-off because masking re-entrance as null
silently hides legitimate user-code bugs (`let r = r; in r.x` would
return null instead of erroring).

## Genuine fix: `ThunkRepr::Promise(Rc<RefCell<NixAttrs>>)`

The empirical bridge above pinpoints the missing piece.  The proper
fix is a new `ThunkRepr` variant carrying a `RefCell` of partially-
constructed `NixAttrs`:

* Outer `let x = f x; in x` (or any self-recursive thunk) detected
  at `eval_let` / `eval_letrec` time — flag the binding.
* When forcing such a thunk, transition `Suspended → Promise(empty)`
  (not `Suspended → Blackhole`).  Run the body.
* As the body evaluates attrset construction, append keys to the
  promise's `NixAttrs` incrementally.
* Inner forces that re-enter the still-evaluating thunk return the
  current state of the promise (an actual `Value::Attrs`, supporting
  `.key` selection, but missing keys still trigger the cycle).
* When the body returns, transition `Promise → Evaluated(final)`.

This matches cppnix's semantics for `lib.fix` / `let x = f x; in x`
and unblocks the rest of `lib.modules.nix` without any sentinel
hacks downstream.  The Promise variant interacts with `!Send` /
`!Sync` correctly (single-threaded evaluator, `Rc<RefCell<>>` for
the cell) and the existing blackhole detection still triggers for
non-attrset-shaped cycles (e.g. `let r = r; in r + 1` still errors
because `Promise` only makes sense for attrset-typed fix points).

## What the partial fix tried (and what it tells us)

Attempted: catch `EvalError::InfiniteRecursion` in `eval_select`
(both at `force_concrete` of the base and at `traverse_attrpath`)
and fall back to the `or default` branch.  Hypothesis: the cycle
flows through `config._module.args.<x> or args.<x>` in modules.nix's
`applyModuleArgsIfFunction`, so catching there would unblock.

Result: **zero hits** during nixosSystem evaluation.  Verified with
an `SUI_M26_TRACE=1` instrument that logged every catch — the
nixosSystem failing path never triggers `eval_select`'s InfRec
fallback.  Conclusion: the cycle bottoms at a sui-internal Native
thunk and re-enters the OUTER let-binding thunk (`declaredConfig`)
through a path that does NOT pass through a `Select-with-default`.
The catch is still landed because it's a correctness improvement
for the unrelated `let x = x; in x.foo or default` pattern (passes
that microtest) — but it is not the M2.6 fix.

The genuine fix is at the Native-thunk re-entrance: when
`mapAttrs` / `zipAttrsWith` / `lib.fix`-driven thunks re-force the
outer fix-point thunk that is currently being computed, sui must
provide the PARTIAL accumulated value (cppnix's behavior for
`let x = f x; in x` re-references), not blackhole with
InfiniteRecursion.  That requires extending the thunk state
machine with a "promise" variant that carries a not-yet-complete
attrset growing as `f`'s body runs.

## Bisection results (what passes vs. what fails)

The cycle is **not** present in any of the building blocks; it
emerges only from the full base-module-set interaction.  Concrete
data points (`sui eval --no-vm --impure --raw …`, nixpkgs pinned to
`b77b3de`):

| # | scenario | sui result |
|---|---|---|
| A | `lib.evalModules { modules = [ { config = {}; } ]; }` → `.config.foo or "ok"` | **`"ok"`** ✓ |
| B | `_module.args = { x = "lazy"; }` then read `_module.args.x` | **`"lazy"`** ✓ |
| C | one module sets `_module.args.greeting`, another destructures `{ greeting, ... }` | **`"hello"`** ✓ |
| E | `specialArgs = { extra = "special"; }` flow to destructured arg | **`"special"`** ✓ |
| F | `eval-config.nix { system; modules = []; baseModules = []; }` | bounded error (missing option) — no infinite recursion |
| H | `baseModules = [ "${nixpkgs}/nixos/modules/misc/version.nix" ]` | bounded error (missing `assertions`) — no infinite recursion |
| I | `baseModules = [ assertions.nix, version.nix ]` (no nixpkgs.nix) | bounded error (missing `environment`) — no infinite recursion |
| J | `baseModules = [ nixpkgs.nix ]` alone | `AttrNotFound("_module")` — no infinite recursion |
| K | `lib.nixosSystem { system; modules = []; }` (baseModules defaults to the full 1982-module list) | **InfiniteRecursion (16-frame chain above)** ✗ |

So:
- the bug requires the FULL baseline-module set
- single modules and small subsets do not reproduce it
- the destructure pattern + `_module.args` extension + specialArgs
  flow are all individually correct in sui-eval
- a strong suspect is `nixos/modules/misc/nixpkgs.nix:358`:
  ```nix
  _module.args = {
    pkgs = lib.mkOverride lib.modules.defaultOverridePriority finalPkgs;
  };
  ```
  where `finalPkgs` is a derivation of `config.nixpkgs.*`.  The
  cppnix merge system resolves priority on the `mkOverride` wrapper
  WITHOUT forcing `content` (`finalPkgs`).  If sui's option merger
  forces `content` while resolving priority for `_module.args.pkgs`,
  the cycle is hit: `finalPkgs` reads `config`, `config` is being
  computed, blackhole.

## Operational repro on rio

```sh
$SUI eval --no-vm --impure --raw '
  let n = builtins.getFlake "github:NixOS/nixpkgs/b77b3de";
  in (n.lib.nixosSystem { system = "x86_64-linux"; modules = []; })
     .config.system.name
'
# → Eval(InfiniteRecursion("force chain (16 frames): …"))
```

Note: this uses `--no-vm` (tree-walker).  The bytecode VM hits the
same blackhole and falls back to the tree-walker, so the cycle is in
the shared module-system / option-merge code path, not in either
engine specifically.
